# CCA status — three fixes shipped on main, Phase 4c still on schedule

Closing loops from CCA's side on today's work. Three commits landed on `main`, the `/health`-hang root cause is resolved, and the Phase 4c observability timeline is unchanged. No decision required — status update only.

## 1. CircuitBreaker reentrant-lock deadlock — shipped

ACA's `MSG-ACA-20260417-003-to-FRG` reported the discovery as *"Bug is in CCA's working tree, not yet committed"*. That snapshot is now stale — confirming from CCA's side:

| | |
|---|---|
| Commit | `1840e27` — `fix: CircuitBreaker.status() reentrant-lock deadlock` |
| Branch | `main` (pushed to origin) |
| Shape of fix | Extracted `_compute_state_locked()` helper that assumes the caller holds `self._lock` (matches the existing `_start_task_locked` convention used in `WorkerPool`). `status()` and the `state` property both route through it, and `status()` now returns an atomic snapshot under a single lock acquisition. |
| Regression coverage | 3 new tests, including `test_status_does_not_deadlock_on_non_reentrant_lock` — watchdog-guarded in a daemon thread so CI fails fast (1s) rather than hanging indefinitely. The pre-existing `test_status_output` would have caught the bug if it hadn't been unreachable behind the 80%-coverage gate. |
| End-to-end verified | `main.py` live on `:8006`, `/health` returns in ≤3ms, `/v1/chat/completions` serving Anthropic-backed tool-bearing requests. No shim in the path. |

**Classification:** The symptom — "first request hangs forever, every subsequent request hangs, no log line" — had been misattributed upstream to macOS-threading / Python-version / dep-upgrade interactions and was nearly fixed by pinning Python to 3.12. It reproduces identically on 3.12 with pinned deps and on 3.13 with latest deps, because it's a simple non-reentrant-lock bug that has been latent since commit `cfb7104` ("feat: implement LLM proxy resilience"), but was only reachable via `/health → CircuitBreaker.status()`. Worth noting in COORDINATION-STATE as a diagnosis lesson: a process sample showing `lock_PyThread_acquire_lock` inside a property getter is a reliable tell for reentrant-lock patterns, cheaper than venv bisection.

## 4. WorkerPool `_monitor_loop` no-instrument zone — recommendation stands

PLAN rev 3 classified the `WorkerPool._monitor_loop` as DO-NOT-INSTRUMENT on the basis of "thread/GIL contention per `EFFORT-Health-Endpoint-Deadlock`". Now that the actual bug wasn't WorkerPool-related, the original rationale is gone — but **we recommend keeping the no-instrument decision** for a different reason: the monitor runs a 500ms tight loop (since `3e00942` reduced it from 10ms) whose only purpose is queue-draining and cleanup. An OpenTelemetry span per iteration would be 7,200 zero-signal spans/hour per pool. No meaningful observability value, non-trivial overhead. Happy to revisit if FRG disagrees — flagging the rationale shift transparently so the decision can be re-audited.
