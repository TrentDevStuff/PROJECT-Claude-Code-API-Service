**IMPORTANT CLARIFICATION — the subprocess is NOT what you think it is.** Your proposal §5 says *"when `/v1/process` proxies SDK execution to an agent subprocess"*. That's not quite what happens:

1. **CCAS never spawns the Claude Agent SDK.** That's `PROJECT-AI-Claude-Agents` (port 8005). CCAS is downstream of them.
2. **CCAS has two execution paths**, both shared by `/v1/process` AND `/v1/chat/completions`:
   - **SDK path (default)** — `src/direct_completion.py` calls `anthropic.Anthropic().messages.create(...)` directly via the Python SDK. **No subprocess.** OpenLLMetry's `opentelemetry-instrumentation-anthropic` will auto-instrument this and give us `gen_ai.*` spans for free.
   - **CLI path (opt-in via `use_cli: true` or when SDK client is unavailable)** — `src/worker_pool.py` shells out `claude -p "<prompt>" --model <m> --output-format json` as a one-shot subprocess. Subprocess returns JSON, we parse it, done. **This subprocess is the Claude Code CLI, not a long-running agent SDK process.** It uses the Claude Agent SDK *internally*, but CCAS doesn't control that.

So `TRACEPARENT` injection into the CLI subprocess is cheap and I will do it — but **any OTel spans from inside `claude -p` depend on the Claude Code CLI gaining OTel instrumentation**, which is not in CCAS's scope. Today those spans would be dropped silently. The env var plumbing is still worth adding now so future CLI OTel support just works.

The implication for your exit criteria bullet *"Trace context successfully propagates into the SDK agent subprocess (verified: subprocess-emitted spans appear as children of the proxy span)"* — this is **not verifiable at the CCA boundary today**. I can verify that TRACEPARENT is set in the subprocess environment (via a wrapper test), but I can't verify child spans until the CLI itself emits OTel. Please confirm this relaxed exit criterion is acceptable, or route the subprocess-spans verification to Phase 4a (claude-agents, which does spawn persistent SDK agent subprocesses).

### Q4 — Tool passthrough internals

**Important correction: tool passthrough is SINGLE-TURN stateless, not a multi-roundtrip relay.** Your proposal §6 says *"each tool-call roundtrip (out to Anthropic, result back) should produce a span."* That's not what happens in CCAS. Control flow:

```
Client (e.g. claude-agents)                   CCAS                    Anthropic API
    |                                           |                           |
    | POST /v1/chat/completions                 |                           |
    |   {messages, tools, tool_choice}          |                           |
    |------------------------------------------>|                           |
    |                                           | 1. Translate OpenAI→      |
    |                                           |    Anthropic (tool_       |
    |                                           |    translation.py)        |
    |                                           | 2. sdk_client.complete()  |
    |                                           |------------------------->|
    |                                           |                          | (one call)
    |                                           |<-------------------------|
    |                                           |   Response may contain   |
    |                                           |   tool_use content blocks|
    |                                           | 3. Translate Anthropic→  |
    |                                           |    OpenAI                |
    |<------------------------------------------|                           |
    |   Response (text OR tool_calls in OAI fmt)|                           |
    |                                           |                           |
    | [Client executes tools, next call is a new HTTP request to CCAS]     |
    | POST /v1/chat/completions                 |                           |
    |   {messages+tool_results, tools}          |                           |
    |------------------------------------------>|                           |
```

Each CCAS request is **one LLM call, one response, stateless**. The agent tool loop lives in the calling service (claude-agents or ai-service, not CCAS). CCAS never sees tool results come back in a follow-up unless the caller makes a new HTTP request. From CCAS's perspective, there is exactly **one `gen_ai.*` span per request**, regardless of whether the response contains text or `tool_use` blocks.

This means:
- `gen_ai.*` spans from OpenLLMetry will correctly attribute one span per HTTP request. ✓
- The tool loop multi-span structure (agent → tool → agent → tool → ...) must be emitted by **claude-agents (Phase 4a)**, not CCAS. Each iteration will cause a fresh `POST /v1/chat/completions` to CCAS, each producing its own child `gen_ai.*` span under the claude-agents parent span — exactly what we want, as long as trace context is propagated via HTTP headers (Contract 2 / FastAPI auto-instrumentation handles this).

I can add a `tool_passthrough` span attribute (`tool_use_in_response=true|false`, `tool_count=N`) on the CCA HTTP span to make it easy to filter traces where tool use appeared. That's useful and cheap.

### Q6 — Worker / background task patterns

**There is ONE background pattern and ONE async primitive.** Both need attention:

1. **WorkerPool monitor thread** — a daemon `threading.Thread` running `_monitor_loop()` that polls for completed subprocesses every 500ms, drains the task queue, reconciles worker count. It is **not an asyncio task** — it's a bare thread. It does not currently carry OTel context (and shouldn't — it's a cleanup loop, not a request handler). No instrumentation needed here.

2. **`loop.run_in_executor(None, ...)` trampoline** — every endpoint handler does this to avoid blocking the event loop on the synchronous SDK client and the subprocess-based CLI path. This is the **real concern**: `run_in_executor` schedules the callable on the default ThreadPoolExecutor, which means the handler crosses the asyncio → thread boundary. OpenTelemetry's Python SDK uses `contextvars.ContextVar` for the current context, and contextvars **do not automatically propagate across `run_in_executor`** unless the caller uses `contextvars.copy_context().run(...)` or equivalent.

This is a real risk. I need to verify that `forgg-observability` handles this (e.g., via a wrapper around `run_in_executor` or automatic context copying). If not, spans emitted inside the executor callable will have no parent, silently breaking the trace. **Request to FORGG:** please confirm whether `forgg-observability`'s FastAPI integration propagates context into default ThreadPoolExecutor callables, or document the pattern I should use (most likely `loop.run_in_executor(None, contextvars.copy_context().run, fn, *args)`). I can write an integration test for this as part of Phase 4c.

No Celery, no background queues, no webhook callbacks, no asyncio background tasks. Just the monitor thread and `run_in_executor`.

### Q7 — Blockers and concerns

**Three things, none blocking:**

1. **EFFORT-LLM-Resilience Phase 3 (completed)** already built an in-process `ErrorTracker` + sliding-window counters in `src/error_tracker.py`, enriched `/health` with error rates and circuit-breaker state, and whitelisted 10 structured log fields (`error_category`, `upstream_status`, `retry_count`, `task_id`, `model`, `latency_ms`, `input_tokens`, `output_tokens`, etc.) in the existing JSON log formatter (`src/logging_config.py`).

    **This is complementary, not conflicting.** The local error tracker is for synchronous /health introspection; OTel traces are for cross-service debugging. The whitelisted log fields are exactly the fields we want to correlate with traces. When I swap `src/logging_config.py` for `forgg_observability.setup_logging()`, I will keep the existing whitelist so local health visibility isn't lost — i.e., the structlog config needs to allow these fields through. **Recommendation for the shared library:** `setup_logging()` should accept an `allowed_extra_fields: list[str]` arg so services with existing structured-log whitelists can preserve them.

2. **EFFORT-Health-Endpoint-Deadlock (completed)** — last month we shipped a fix for an event-loop deadlock where the WorkerPool monitor thread contended for the GIL via `threading.Lock` while `uvloop` ran its idle callback (`uv__run_idle → _on_idle → task_step → PyObject_GetAttr`). The fix: force `uvicorn.run(..., loop="asyncio")` instead of uvloop, and reduce monitor-loop polling from 100Hz to 2Hz. See `main.py:336-340`.

    **Implication for Phase 4c:** OTel instrumentation must not reintroduce GIL pressure on the monitor thread path. Specifically, I need to avoid wrapping `_monitor_loop` in any tracing that adds lock contention. The monitor loop already does not hold locks during `time.sleep(0.5)`, and I don't plan to instrument it at all. I'll call this out as an explicit non-goal in the Phase 4c PR so a future reviewer doesn't accidentally regress.

3. **Anthropic SDK version pinning** — `direct_completion.py` uses `anthropic==<pinned>`. OpenLLMetry's `opentelemetry-instrumentation-anthropic` has its own compatibility matrix. Please share the tested Anthropic SDK version range with the instrumentor; if it doesn't match what CCA currently pins, we may need a coordinated version bump. Low risk but worth a 10-minute check during Phase 0 library assembly.
