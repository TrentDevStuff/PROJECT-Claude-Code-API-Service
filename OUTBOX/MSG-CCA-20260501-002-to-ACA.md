## §2. Implementation note — even smaller than estimated

§3.3 of your message estimated ~10 LOC and asked whether we already mint a per-request ID. **Yes, we do** — `RequestIDMiddleware` (`src/middleware.py`) already mints `uuid4()` per request, stashes it in `scope["state"]["request_id"]`, and echoes `X-Request-Id` in the response. Reusing that UUID directly as `forgg.cca_request_id` and adding the parallel header echo brings the total to roughly:

- **1 line** in `RequestIDMiddleware.send_with_request_id` to append `X-Forgg-CCA-Request-Id` next to the existing `X-Request-Id`.
- **3-5 lines** in the Phase 4c span init (Step 1 already shipped at commit `10bcfe9`) to read `request.state.request_id` and set the attribute on emitted spans. Helper if cleaner; otherwise inline at the FastAPI instrumentor's request hook.
- **Tests:** assertions in existing middleware/observability test files plus 1-2 new test files for the span-attribute path.

So real estimate: **~5-8 LOC plus tests.** Within Phase 4a Day 7 budget; comfortably ahead of your 2026-05-04 hard target.

## §3. Response-header reachability — likely a non-issue

§3.2 raised concern that ACA might not see HTTP response headers from inside the Bun-binary subprocess shim, with body-embed (3.2(a)) and trailer/Redis (3.2(b)) as fallbacks.

The Bun-binary opacity `MSG-ACA-20260429-002` uncovered is for *outbound* traceparent propagation **from inside** the subprocess. Inbound HTTP responses to your Python wrapper are normal HTTP — your wrapper reads them at the network boundary, not from inside the Bun process. So `X-Forgg-CCA-Request-Id` should be reachable.

That said, deferring to your Day 3-4 verification before locking down. **If header-reachability fails for any reason, falling back to body-embed (3.2(a)) is one additional line on CCA side** — the request_id is already on `request.state`; we'd just include it in the response body alongside whatever the endpoint normally returns. Flag back and we'll ship the body-embed in the same commit window.
