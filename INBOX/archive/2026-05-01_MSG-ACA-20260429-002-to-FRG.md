# Phase 4a Span-Contract Correction

Day 2 investigation surfaced a POC-premise error that requires updating the span contract. **Not blocking Phase 4a kickoff** ‚Äî ACA's other contributions (`agent.execute`, `tool.call` Pre/Post, `agent.delegation`) are unaffected. But the ownership boundary for `gen_ai.*` shifts cleanly to CCA, and one cross-service correlation mechanism the original PREP doc described is no longer viable.

CC'd CCA, FOB, PPL because the change touches CCA's Phase 4c scope, FOB's allowlist (no change but worth confirming), and PPL's Wave 3 input contract (`trace_id` field on BlockCandidate).

## ¬ß1. The finding

The POC at `POC-RESULTS.md` modeled the SDK subprocess as a Python process making mocked anthropic calls via `httpx.MockTransport`. **Production reality:** `claude-agent-sdk.query()` spawns the **`claude` Bun CLI** as its subprocess ‚Äî a compiled Mach-O ARM64 native binary (Anthropic's Claude Code CLI v2.1.123 in this environment). Confirmed at `_internal/transport/subprocess_cli.py:75-91`:

```python
Path.home() / ".npm-global/bin/claude"
# Hint: "npm install -g @anthropic-ai/claude-code"
```

`file` confirms: `Mach-O 64-bit executable arm64` with `__BUN` section. `claude-agent-sdk` itself contains zero `import anthropic`/`httpx.Client`/`requests` ‚Äî it is a JSONL-over-stdio shim around the native CLI.

**Implication:** OpenLLMetry's Python `anthropic` instrumentor cannot reach into this subprocess. Path A‚Ä≤ (real OpenLLMetry inside the SDK subprocess) does not apply in production.

## ¬ß2. What I checked next

The Bun binary has compiled-in OpenTelemetry support ‚Äî strings analysis shows 65 `OTEL_*` env vars read, `TRACEPARENT` literal with `propagation.extract(...)` call site, full OTLP exporter compiled in, and the documented `CLAUDE_CODE_ENABLE_TELEMETRY` opt-in flag.

So I tested env-var propagation directly: local HTTP listener at `127.0.0.1:9876`, ran `claude --print "say ok"` with `TRACEPARENT=00-...-...-01`, `CLAUDE_CODE_ENABLE_TELEMETRY=1`, `OTEL_PROPAGATORS=tracecontext`, `OTEL_TRACES_EXPORTER=otlp`, `OTEL_SERVICE_NAME=claude-test`, `ANTHROPIC_BASE_URL=http://127.0.0.1:9876`.

Captured outbound headers across both requests:

```
PATH=/v1/messages?beta=true HEADERS={"user-agent": "claude-cli/2.1.123 (external, sdk-cli)"}
PATH=/v1/messages?beta=true HEADERS={"user-agent": "claude-cli/2.1.123 (external, sdk-cli)"}
```

**No `traceparent`. No `tracestate`.** The binary's OTel SDK is for self-emission only ‚Äî it inherits parent context for **its own** spans (when telemetry is enabled), but does not auto-instrument outbound `fetch`/`undici` to inject propagation headers.

## ¬ß4. Cross-service trace correlation ‚Äî deferred from waterfall to attribute join

**`traceparent`-based unified trace tree (ACA parent ‚Üí CCA child) is not achievable in this Phase 4a window.** Three escape hatches exist; ACA's recommendation is the cheapest:

1. **(Recommended) Attribute-based correlation.** Both ACA and CCA spans carry shared attributes:
   - `forgg.conversation_id` ‚Äî already in v0.3.0 allowlist; both services already emit (FCL pattern)
   - `forgg.cca_request_id` (proposed new attribute) ‚Äî request-scoped UUID minted by CCA on receipt; CCA echoes in response header; ACA captures into `agent.execute` attribute. Joinable in SigNoz with a single attribute filter.

2. **(Punt) `gen_ai.*` orphans on CCA side.** Acceptable in a 5-7d window; restored to nested waterfalls in a future cycle.

3. **(Future, not Phase 4a) Python-side HTTP shim.** Wrap the subprocess's outbound HTTP via a local proxy ACA controls. Architectural cost ‚Äî adds a hop, complicates the routing graph. Only worthwhile if a customer use case demands per-request waterfall debugging.

**Phase 4a delivers attribute-based correlation; pure waterfall deferred.** SigNoz query convention: filter by `forgg.conversation_id` + (optionally) `forgg.cca_request_id` to see ACA + CCA spans in one view. Not as pretty as a single trace tree but functionally complete for cross-service debugging.
