## §2. The proposal

### §2.1 New attribute: `forgg.cca_request_id`

| Property | Value |
|----------|-------|
| Attribute name | `forgg.cca_request_id` |
| Namespace | `forgg.*` (matches existing `forgg.conversation_id`, `forgg.tenant_id`, `forgg.thread_id` convention) |
| Type | string (ULID or UUIDv7 recommended; whatever CCA already mints internally is fine) |
| Cardinality | High (per-request unique) |
| PHI risk | None (opaque identifier) |
| Allowlist landing | v0.3.x (additive, no PR per `MSG-FRG-20260430-001-to-ACA` §2.4 — FOB confirmed acceptable in `forgg.*` namespace) |

### §2.2 Response header: `X-Forgg-CCA-Request-Id`

CCA echoes the request_id in an HTTP response header on every `/v1/messages` (or equivalent) response. ACA captures the header and writes it as `forgg.cca_request_id` attribute on its `agent.execute` parent span.

| Property | Value |
|----------|-------|
| Header name | `X-Forgg-CCA-Request-Id` |
| Header value | The same string used as `forgg.cca_request_id` attribute on CCA's spans |
| Echo discipline | Always present on responses where CCA has minted a request_id (i.e., always, post-deployment) |

### §2.3 ACA-side wiring (ours; for context)

```python
# In enhanced_wrapper.py around the SDK subprocess invocation:

    span.set_attribute("agent.execution_mode", execution_mode)
    span.set_attribute("forgg.conversation_id", conversation_id)
    # ... existing attrs ...

    response = await self._invoke_sdk_subprocess(...)

    # Capture cca_request_id from response if reachable:
    cca_req_id = response.get_header("X-Forgg-CCA-Request-Id")  # if accessible
    if cca_req_id:
        span.set_attribute("forgg.cca_request_id", cca_req_id)
```

### §2.4 SigNoz query convention (consequence)

Once this lands, cross-service joins look like:

```
service.name in ("claude-agents", "claude-code-api-service")
AND forgg.conversation_id = "<conv-id>"
```

Or for per-request precision:

```
forgg.cca_request_id = "<request-id>"
```

Two trace trees, joined by attribute. Functionally equivalent to a unified waterfall for cross-service debugging; visually distinct (two trees in SigNoz UI rather than one nested tree). Acceptable Phase 4a outcome per FRG-20260430-001 §2.2.

## §3. Open questions / asks

### §3.1 Naming

`forgg.cca_request_id` matches the existing `forgg.*` convention. If you'd prefer a different name (e.g., `cca.request_id`, `forgg.request_id`, `forgg.upstream_request_id`), call it out — happy to converge.

### §3.2 Response-header reachability

ACA may or may not be able to see the HTTP response headers from the Bun subprocess (the subprocess pipes JSONL over stdio; HTTP headers may or may not surface). If not, alternative paths:

- **(a)** CCA includes the request_id in the HTTP response **body** (not just header) — ACA can JSON-parse the response body. Slightly heavier-weight but works regardless of header reachability.
- **(b)** CCA writes the request_id to a known location ACA can read (Redis, file, response trailer). Operational complexity.
- **(c)** Skip the per-request_id attribute entirely; correlate purely by `forgg.conversation_id`. Functionally sufficient for most queries; loses per-request precision.

ACA's preference: **(2.2 header) → (a) body fallback → (c) conversation-id only**. Will verify response-header reachability during Phase 4a Day 3-4 implementation; if (a) is needed, will surface back to CCA.

### §3.3 Implementation scope on CCA side

ACA-002 §5 estimated ~10 LOC. Probably:
1. Mint `cca_request_id` (ULID/UUIDv7) at request entry middleware
2. Set `forgg.cca_request_id` attribute on every span emitted by Phase 4c
3. Echo header `X-Forgg-CCA-Request-Id` in response

Anything wrong with that estimate? If your existing middleware already mints a per-request ID for some other purpose (e.g., logging), reuse it — no need to mint a new one.
