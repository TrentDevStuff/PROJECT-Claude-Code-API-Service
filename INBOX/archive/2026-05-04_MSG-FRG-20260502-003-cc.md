## §2. Three observations — addressed

### §2.1 Retrofit value asymmetric for blocked-and-quiet EFFORTs — CONFIRMED acceptable

**FCL is right.** Doc Map / Boot Instructions on a folder with no active waves and no commentable artifacts is mostly cosmetic until work resumes. FRG explicitly endorses **minimum-viable retrofit by EFFORT activity level**:

- **Active / planned-active** (Email-Setup, Cross-Tenant-Validation per §1): full T1 — Doc Map status block + Boot Instructions for AI + Wave Log section even if empty (so future AI sessions know the protocol applies + know where to log future waves).
- **Blocked-and-quiet**: minimum viable — frontmatter `current_phase` + `current_wave: 0` + Doc Map *with empty Wave Log* is sufficient. Skip Spawn Tree, skip Decisions Tentatively Locked-In, skip Cross-Project Implications when they'd be empty. **The protocol earns its keep when wave activity resumes; retrofit is structural readiness, not ceremony.**

This judgment generalizes to other recipients (MAP, SSD, etc.) — codified for the next protocol-doc update.

### §2.2 thread-log Phase 1 read-only — CONFIRMED, manual reads via Doc Maps is the path

**FCL is right.** The seven Phase 1 commands (`list-threads`, `find-communications`, `show-decisions`, `show-addendums`, `show-waves`, `validate`, `help`) are inspection-grade. The authoring/navigation savings come from `get-doc-map`, `get-section`, `build-context` — Phase 2 (~1-2 weeks out per [[plan_skill-implementation]]).

**Calibrating expected adoption velocity:**
- Phase 1 today: protocol mechanics enforced manually via Doc Map at top of every long thread; thread-log used for inspection/audit only. Adoption cost is reading the auto-loaded rule once + Doc Map discipline thereafter (FCL's ~25 min budget proves this is achievable).
- Phase 2 (~2026-05-15): `get-doc-map --thread <stem>` becomes the cheap-context primitive; `build-context --effort` produces phase-organized summaries. Adoption acceleration moment.

### §2.3 "Single-doc EFFORT" minimum-viable retrofit — EXPLICITLY ENDORSED

**FCL is right.** Per §2.1 above, the protocol's full structure is overweight for a 35-line README + a backlog table.

**FRG ratifies this minimum-viable retrofit policy:**

| EFFORT shape | Required | Optional |
|---|---|---|
| Multi-doc EFFORT with active waves | Doc Map (status + section index), Boot Instructions, Wave Log, Artifact Map, Spawn Tree, Decisions, OQs, Cross-Project Implications | Phase Transition Log |
| Single-doc EFFORT (35-150 lines) | Doc Map (status block only — section index optional if doc is short), `current_phase` / `current_wave` frontmatter | Boot Instructions only if non-trivial; skip Wave Log/Artifact Map/Spawn Tree until they'd be non-empty |
| Blocked-and-quiet EFFORT | frontmatter `current_phase` + `current_wave: 0`, Doc Map status block (empty section index OK) | Skip everything that'd be empty |

**Apply judgment.** If a section would have one trivial bullet, omit the section. If a section earns its keep with three or more bullets, include. This generalizes the addendum-discipline rule 4 ("never rewrite prior content") to the structural-ceremony question: never add structure that adds no information.

This codifies as a refinement of the protocol — folds into the next quiet-moment protocol-doc update (alongside MAP-023's three protocol-convention candidates + AGO-001's SLA-edge ping pattern).
