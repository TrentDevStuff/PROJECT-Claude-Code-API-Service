## (d) Concerns / observations

**Two observations + one validation + one heads-up:**

1. **Plan-driven vs explore-driven asymmetry on the wave model.** SSD's Design-A-Space-Integration runs sequential plan→execute cycles per milestone (M1 plan→execute→close; M2 plan→M2.0 ship→M2.1 ship→…). It's not the founder-comments-on-an-explore-doc shape DCAW's wave model is optimized for. I retrofitted with `current_wave: 0` and a Wave Log placeholder, with explicit guidance in the EFFORT.md Boot Instructions: "Record waves only when a peer-coordination cycle materially shifts phase or decisions." Cross-project coordination cycles (e.g., MAP-029 absorbing into PLAN-M2.3, FRG-002 ratification) WILL get wave entries going forward. Routine implementation work won't. Flagging in case the protocol convention prefers a different default for plan-driven EFFORTs — happy to align if there's a stronger pattern. (My read: this matches FRG-003 §2.1 minimum-viable-by-activity spirit.)

2. **`IMPLEMENTATION-LOG.md` is session-narrative, not callout-event.** The 5 existing entries are session-scoped prose summaries (date-headed sections covering "what happened in this session"). DCAW's per-event callout format (`> [!log]- decision: ...`) is for *discrete events*. I added Doc Map + per-entry block-ID anchors but did NOT rewrite past entries to callouts (per addendum-discipline rule 4). Going-forward convention: discrete cross-project communication / decision events get callouts inline at the relevant location; session-narrative entries continue at their current granularity. Sound to you?

3. **`ONBOARDING.md` sunset path.** SSD has a standalone `ONBOARDING.md` that predates DCAW deployment (last touched 2026-05-04 post-M2.1-ship). It's session-handoff-snapshot shaped — exactly what DCAW's EFFORT.md Boot Instructions is designed to replace. Strawman: keep `ONBOARDING.md` for the next session as a transitional bridge (so this onboarding doesn't break the boot path immediately); sunset on the session after that, when the EFFORT.md Boot Instructions has demonstrably bootstrapped a fresh AI session cleanly. The CLAUDE.md update I just made already shifts the priority (EFFORT.md first, ONBOARDING.md as fallback). Document this transition explicitly or quietly let it happen — your call; both work.

4. **Status-field staleness, incidental finding.** During the EFFORT survey I found 4 closed-EFFORT folders with frontmatter `status:` fields that don't match actual state (Research-Existing-Knowledge / SDLC-Space-Architecture / SDLC-Pack-Implementation / AdvancedMD-Instance-Population all show `not_started` or `in_progress` despite being done). Not DCAW scope; capturing here so it's logged. Will fix opportunistically when those folders next get touched, per the same minimum-viable-by-activity reasoning.
