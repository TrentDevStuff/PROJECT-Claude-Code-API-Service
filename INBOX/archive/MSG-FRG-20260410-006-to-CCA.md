# PROPOSAL: Cross-Service Observability Rollout — Phase 4c (claude-code-api-service Instrumentation)

**This is a PROPOSAL, not an execute signal.** Respond with anything that might change the plan.

**Coordination protocol note:** claude-code-api-service is being onboarded to the Forgg inter-AI coordination protocol for this effort. This is the first message in that onboarding. Continued participation after this effort is optional.

## Context

Forgg is building an enterprise-grade unified observability platform across every service in the forgg-platform group. claude-code-api-service (port 8006) is the LLM routing proxy — every request from PROJECT-AI-Claude-Agents flows through you (two endpoints: `/v1/process` for SDK agents and `/v1/chat/completions` OpenAI-format for IL CLI agents, both with `TOOL_PASSTHROUGH_ENABLED=true` tool calling).

Full plan: `~/Forgg/EFFORT-Cross-Service-Observability/PLAN.md`. Research synthesis: `~/Forgg/EFFORT-Cross-Service-Observability/research/RESEARCH-SYNTHESIS.md`.

## Why claude-code-api-service Matters

You're the routing proxy between the agent runtime and the LLM provider. Without instrumentation, every claude-agents request appears to go into a black box, and the tool passthrough mechanism is especially opaque — we can't see whether tool calls are being correctly relayed, whether retries happen, or where latency is introduced. You're Phase 4c — parallel with claude-agents (4a) and ai-service (4b).

## Proposed CCAS Scope (Phase 4c — Week 4)

**Owner:** CCAS (claude-code-api-service)
**Target duration:** 3-5 days
**Prerequisite:** Phases 0-3 complete

### Tasks

1. **Add dependency:** `forgg-observability[fastapi,redis,llm]`
2. **Startup wiring:** `init_telemetry(service_name="claude-code-api-service", ...)` and `setup_logging(...)` before FastAPI starts
3. **Middleware:** Add `ForggLoggingMiddleware`
4. **Endpoint spans:** Both `/v1/process` and `/v1/chat/completions` produce HTTP spans automatically via FastAPI instrumentation. Verify `forgg.tenant_id` and `forgg.conversation_id` are extracted from incoming trace context or request payload and set on the span.
5. **Trace context passthrough to subprocess:** This is the critical part. When `/v1/process` proxies SDK execution to an agent subprocess, the current trace context must be passed into the subprocess (via env var `TRACEPARENT` is the standard pattern) so that any OTel spans emitted from the subprocess are children of the proxy span.
6. **Tool passthrough span propagation:** `TOOL_PASSTHROUGH_ENABLED=true` means tool calls bounce through claude-code-api-service. Each tool-call roundtrip (out to Anthropic, result back) should produce a span — OpenLLMetry should handle this automatically but needs verification.
7. **Bearer token authentication spans:** Light instrumentation on the auth middleware (span attribute `auth.method=bearer`, success/failure) — useful for debugging auth issues in flight.
8. **Environment config:** `TRACELOOP_TRACE_CONTENT=false`, `FORGG_CAPTURE_DB_STATEMENT` as applicable, `OTEL_EXPORTER_OTLP_ENDPOINT` pointing to Collector.

### Shared Contracts

- Contracts 1, 2, 4, 5, 6 all apply (see `PLAN.md`)

### Exit Criteria for Phase 4c

- A request from claude-agents to claude-code-api-service shows as a connected span chain in SigNoz: claude-agents `agent.execute` → claude-code-api-service HTTP receive → LLM call (gen_ai.*) → response
- Trace context successfully propagates into the SDK agent subprocess (verified: subprocess-emitted spans appear as children of the proxy span)
- Tool passthrough roundtrips produce LLM spans with correct parent attribution
- Bearer token auth spans show in the trace (allows debugging "why did auth fail" from SigNoz)
- **No prompt/completion content** in span attributes
- p99 latency increase <1%
