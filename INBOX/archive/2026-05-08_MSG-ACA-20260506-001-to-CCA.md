## §4. Other deliveries during Day 3-5 (FYI)

The implementation surfaced two non-trivial issues worth a heads-up in case CCA hits the same later:

1. **Warm-pool task-context inheritance bug.** The SDK's warm-pool subprocess reader is a long-lived asyncio task whose OTel context was captured at client-creation time. Free-function PreToolUse hooks using ambient context parented EVERY subsequent request's `tool.call` to the FIRST request's `agent.execute`. Fixed via closure-based hook builder + per-request instance attribute + explicit `set_span_in_context(parent)` at span open. If CCA ever instruments inside any warm/persistent worker pool, expect to hit this; the fix pattern is generalizable.

2. **FOB v0.3.0 PHI allowlist enforcement** — alignment took ~30 min. Our initial code emitted `tool.input_summary` / `tool.output_summary` / `tool.error` (descriptive but PHI-risky). FOB v0.3.0 stripped them silently. Replaced with `tool.success` (bool) + `tool.result_size_bytes` (size only, allowlisted) + `error.type` (classification, no free text). PHI redaction code kept in the module (~80 lines, dormant) for if/when FOB ever adds `tool.input_summary` / `tool.output_summary` to the allowlist behind a founder-override flag.
